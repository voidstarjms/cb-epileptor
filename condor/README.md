# HTCondor Sweep Submission

This folder contains everything needed to run the CE vs X0 parameter sweep on the WWU cluster.

## Files

- `wrapper.sh` — HTCondor executable. Activates the venv and runs python.
- `setup_condor.sh` — Generates `condor.sub` with correct absolute paths for this machine.
- `logs/` — Condor output/error logs, organized by run number.
- `condor.sub` — Generated by `setup_condor.sh` (not committed to git).

## Prerequisites

- The venv must be set up at `SR-CB/.cn_venv/` with all dependencies installed.
- You must be on the lab machine (paths in `condor.sub` are machine-specific).

## How to Run a Sweep

All commands should be run from the `cb-epileptor/` project root unless otherwise noted.

**1. Pull the latest code**
```
git pull origin synchronicity_plotting
```

**2. Generate the parameter list**
```
cd src
python generate_params.py
cd ..
```
Creates `src/params_list.txt` with 64 CE/X0 pairs (8x8 grid).

**3. Generate condor.sub and the log folder**
```
bash condor/setup_condor.sh
```
This determines the run number, creates `condor/logs/{run_num}_synchrony/`,
writes `src/current_run.txt`, and generates `condor/condor.sub`.

**4. Test one job manually**
```
cd src
python run_single_sim.py --ce 0.25 --x0 -3.5
cd ..
```
Confirm it completes without errors and that a `.pkl` appears in `src/data/results/`.

**5. Submit all 64 jobs**
```
condor_submit condor/condor.sub
```
Jobs will initially appear as idle (`I`) — this is normal. Condor is queuing them
and will assign machines as they become available. Monitor with:
```
condor_q
```
Jobs transition from idle (`I`) to running (`R`) and disappear when complete.

**6. Generate heatmaps**
```
cd src
python aggregate.py
```

Output files all share the same run number, e.g. for run 1:
- `src/figures/1_synchrony_chi_mean.png`
- `src/figures/1_synchrony_chi_sd.png`
- `src/figures/1_sweep_debug/` — per-simulation LFP + raster plots
- `condor/logs/1_synchrony/` — Condor stdout/stderr logs

## Subsequent Runs

Repeat from step 2. The run number increments automatically each time
`setup_condor.sh` is run, so previous outputs are never overwritten.

## Debugging a Failed Job

If a job fails, check its error log:
```
cat condor/logs/{run_num}_synchrony/err_{process_num}.log
```
